# invoice-pdf-generator

## このプロジェクトが解決する問題

ECサイト運営の実務 PM から、見積書 PDF がダウンロードできる機能の要望があった。
実務では PHP + 既存税計算 API + DOMPDF で実装予定だが、本プロジェクトでは TypeScript + Next.js + Playwright で再現する。

| # | 目的 | 実務との対応 |
|---|------|--------------|
| 1 | PDF 生成 | DOMPDF → Playwright |
| 2 | 税計算 | 既存 API + PHP 加工 → Server Actions による自前実装 |

参考: [ギフトモール 商品ページ](https://giftmall.co.jp/giftcXrCLD/) — 商品詳細ページで購入オプションを入力し、計算結果を見積書 PDF として自動発行する。


## 要件

### 前提

- EC サイトは 8%（軽減税率）・10%（標準税率）など **複数の税率区分を持つ商品** を扱う
- 見積は **商品詳細ページ単位**（1 見積 = 1 商品）。有料オプションにより **1 枚の見積内で税率区分が混在しうる**
- 税計算は **2 レイヤー** で行う（詳細は「設計上の重要な判断」参照）
  1. **売価** — 税込単価 × 数量（EC 表示価格は税込）
  2. **伝票** — 税率区分ごとに税込合計を集計し、区分合計から内消費税を逆算（端数処理は区分合計に対して 1 回）

### ユーザーフロー

```
商品詳細ページで入力 → 税計算 → 即 PDF 生成・ダウンロード
```

- 下書き保存・確定操作・見積履歴 UI は **提供しない**（ギフトモールの自動発行 PDF と同様）
- 再見積が必要な場合は **新規発行** する

### 見積書 PDF

**含める**

- 見積番号・見積日・有効期限
- 商品名・詳細・商品コード・税率・数量・単価・金額（明細表は 1 行）
- 税率区分ごとの税込合計・内消費税
- 有料オプションの説明（表外注記）
- 送料・在庫・価格変動等の備考
- 発行元（自社）情報・インボイス登録番号

**含めない**

- 取引先名・宛先会社情報（拝啓等の定型文のみ）
- 複数商品（別 SKU）を 1 枚にまとめた一括見積

### スコープ外

| 項目 | 理由 |
|------|------|
| 取引先名の記載 | 自動発行 PDF は宛名なしの定型文形式 |
| 取引先マスタ・ログイン | B2B ポータルではなく EC カスタマー向け機能 |
| 複数商品の一括見積 | 法人営業フロー相当。別途問い合わせが前提（拡張予定参照） |

### データ

| 種別 | 保存先 | 用途 |
|------|--------|------|
| 商品マスタ | JSON 等の静的データ | SKU・単価・税率・オプション定義。画面の選択肢 |
| 発行ログ | PostgreSQL + Prisma | サーバー側のみ。カスタマー向け UI は提供しない |

**発行ログの用途**

1. **問い合わせ対応** — 見積番号から発行時点の内容（数量・オプション・金額）を確認
2. **価格・在庫トラブルの説明責任** — 発行時点のスナップショットを根拠に、価格改定・欠品時の説明

発行ログには、見積番号・発行日時・有効期限・商品スナップショット・税計算結果・選択オプション等を保存する。


## 技術スタック
- Next.js
  - フロントとサーバーサイドを TypeScript で一気通貫で書ける。TypeScriptでBFFを開発するという構成に最適。
- Playwright
  - PDF生成速度とCSS表現力のバランスから。比較対象として Puppeteer があるが、ウォームプール使用時のPDF生成速度が Playwrightが3ms、Puppeteerが48ms という差があるため Playwright を選択。
- PostgreSQL + Prisma
  - TypeScript との型統合が自然。

## 設計上の重要な判断

### 税計算

EC 表示価格は税込。税込単価は売価として整数円で確定しており、1 個あたりの内消費税を整数で割り出す必要はない。

**レイヤー 1 — 売価（税込金額）**

- 税込単価 × 数量、および有料オプションの税込加算額を算出する
- 商品・オプションごとに消費税を端数処理しない

**レイヤー 2 — 伝票（インボイス記載）**

- レイヤー 1 の税込金額を税率区分（8% / 10% 等）ごとに合算する
- 区分合計から内消費税を逆算し、端数処理は区分合計に対して 1 回のみ行う
- PDF には区分ごとの「税込合計」「内消費税」を記載する（単価ごとの内税内訳は載せない）

**内消費税の計算式**

税込 = 税抜 × (1 + 税率/100) より、税率区分の合計から次式で逆算する。

```
内消費税 = floor(区分合計 × 税率 ÷ (100 + 税率))
```

例: 区分合計 1,000 円（税込単価 1,000 円 × 1 個）

| 税率 | 途中計算 | 内消費税 |
|------|----------|----------|
| 8% | 1,000 × 8 = 8,000 → 8,000 ÷ 108 = 74.074… | **74 円** |
| 10% | 1,000 × 10 = 10,000 → 10,000 ÷ 110 = 90.909… | **90 円** |

8% の場合の PDF 記載例: 8%対象(税込) 1,000 円 / 内消費税 74 円 / 合計(税込) 1,000 円

**方針**

- 税込合計を正とし、区分合計から逆算した内消費税を記載することで、売価表示とインボイス表示の矛盾を避ける
- 見積 PDF は 1 商品 1 行表示だが、税計算モジュールは商品・オプションの税率に依存しない汎用実装とする（請求書転用を見据える）

**税抜順算を採用しなかった理由**

税抜単価 × 数量 → 消費税（端数処理）→ 税込、という順算も算定方法の一つだが、本プロジェクトでは採用しない。

| 観点 | 税抜順算 | 税込逆算（採用） |
|------|----------|------------------|
| EC 表示 | 税込単価を税抜に戻す必要があり、単位ごとに端数が発生する | 表示どおり税込単価 × 数量で合計が確定する |
| 金額の一致 | 税抜切捨て等により、画面表示と PDF 合計が数円ズレうる | 税込合計を正とするため、画面と PDF が一致する |
| インボイス表示 | 区分合計方式と組み合わせると二重に端数処理が絡む | 税率区分の税込合計から内消費税を 1 回逆算し、PDF の表示単位と一致する |
| 参考実装 | — | 上記 8% の例（ギフトモール見積 PDF も同型） |

例（8%・税抜順算との比較）: 税込単価 1,000 円 × 1 個で、税抜単価を切捨て（925 円）→ 925 + floor(925 × 0.08) = **999 円** となり EC 表示と 1 円ズレる。

### PDF 生成

- コールドスタートを避け、ブラウザを事前起動したウォームプールで使い回す
- ヘッドレスブラウザはメモリを消費するため、同時実行数に上限を設けキューで制御する

### 見積のライフサイクル

- 入力と同時に発行・確定。発行後の更新は行わない（変更は新規発行のみ）
- 有効期限（`validUntil`）により期限内 / 期限切れを判定する。下書き・確定・失効の状態遷移は持たない

### 永続化

- カスタマー体験はステートレス（即 PDF）に近いが、発行ログのみ DB に残す
- PDF 自体がカスタマー向けスナップショット。DB は運用・問い合わせ対応用の正本
