# invoice-pdf-generator

## このプロジェクトが解決する問題
ECサイト運営の実務のPMから見積書PDFがダウンロードできる機能の要望があった。
実務では PHP + 既存税計算API + DOMPDF で実装予定だが、このプロジェクトでは
TypeScript + Next.js + Playwright で実装する。目的は2つ。

1. PDF生成: PHP の DOMPDF に相当する処理を Playwright で実装する
2. 税計算設計: 既存 API 呼び出し + PHP 加工を税計算ロジックを内包した Next.js Server Actions を自前で設計実装する

## 技術スタック
- Next.js
  - フロントとサーバーサイドを TypeScript で一気通貫で書ける。TypeScriptでBFFを開発するという構成に最適。
- Playwright
  - PDF生成速度とCSS表現力のバランスから。比較対象として Puppeteer があるが、ウォームプール使用時のPDF生成速度が Playwrightが3ms、Puppeteerが48ms という差があるため Playwright を選択。
- PostgreSQL + Prisma
  - TypeScript との型統合が自然。

## 設計上の重要な判断
- 税計算ロジック(なぜ明細行ごとの端数処理をしないか)
  - 将来的に請求書と一致されること見越してインボイス制度の端数処理ルールに対応した計算ロジックにする
- ブラウザインスタンスプールとキュー制御
  - PDF生成速度がもっとも重視する指標になるためコールドスタートではなく
あらかじめブラウザを複数起動した状態で待機されておき、リクエストが来たらそのブラウザを使い回す（ウォームプール）
  - ヘッドレスブラウザを利用するためメモリが枯渇しやすい。同時リクエスト数に上限を持たせ、キュー制御でプログラムのクラッシュを防止する
- 見積書のライフサイクル
  - 見積書は発行後に内容を撤回することが商慣習上認められていない。そのため「下書き」と「確定(発行済み)」という状態を管理し、確定後の金額変更を禁止する設計とする
